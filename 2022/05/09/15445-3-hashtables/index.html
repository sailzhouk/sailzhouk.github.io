DB中的数据结构-哈希表

Posted on 2022-05-09 Edited on 2022-05-10 Views:

DB中通过合适的数据结构来组织不同部分的DB数据（e.g. 内部元数据，表核心数据，表索引数据，临时数据等），其设计决策的核心在于：

  • 相应数据结构在底层存储上的具体布局结构
  • 相应数据结构如何处理并发场景

Hash table是一种在DB中常用的数据结构，这一节课对不同种类哈希表的原理以及其优缺点进行了介绍。

Hash Table

Concepts

Hash table无序地存储若干组键值对，对于每一个指定的key通过一个Hash Function将其映射到一个数组对应的offset上，从而在平均O(1)的时间复杂度下获取到查询的value。

Hash table具有O(n)的空间复杂度，和平均O(1)的时间复杂度，时间复杂度最高可至O(n)。

Design Decision

  1. Hash Function
    • Trade-off between being fast vs. collision rate.
  2. Hash Scheme
    • 发生哈希碰撞后的处理策略，增加耗费资源与增加处理逻辑复杂度之间的权衡

Hash Scheme

哈希策略分为静态哈希和动态哈希两种，静态哈希的slots数目固定，存储数目过多时需要一次性对容器进行扩容，一般为原有大小的两倍。容器的初始大小一般最好为预计的条目数的两倍，避免因为冲突降低性能，静态哈希适用于预先知道哈希表中元素大概总数的情况，否则就有可能需要扩容/缩容，影响性能。

动态哈希策略可以动态地调整哈希表的大小。

Static Hash Scheme

  • Linear Probe Hashing
  • Robin Hood Hashing
  • Cuckoo Hashing
Linear Probe Hashing
  • 插入：通过hash函数计算出offset，若对应offset为空，则直接插入，否则依次查询后续位置是否为空，直到找到一个空位置插入
  • 查询：通过hash函数计算出offset，依次查询，直到遇到一个空位置或回到初始位置，若还未找到则查询失败
  • 删除：查询到对应位置进行删除，直接删除会对查询策略产生影响，解决思路有两种：
    • 在删除位置放置一个墓碑（Tomb Stone）
    • 删除后，移动后续位置的元素，填补空出来的位置

对于non-unique key的处理：

  • 将多个value值以链表的形式表示
  • 直接存储重复key
Robin Hood Hashing

名字源于“罗宾汉”，对应这个策略的核心思路：劫富济贫

  • 跟踪每个key与其最佳位置（Hash函数直接计算出来的位置）之间的距离
  • 插入时根据距离大小交换插入位置，距离大的优先级更高（劫富济贫）
CUCKOO Hashing

用多个Hash Table，每个Hash Table设置不同的Hash Seeds

插入时，找任意一个table中可用的位置进行插入，若无可用位置，则删除一个表中对应位置的元素，进行重哈希

Dynamic Hash Scheme

  • Chained Hashing
  • Extendible Hashing
  • Linear Hashing
Chained Hashing

以Buckets List的形式存储hash(key)对应的值

Extendible Hashing
  • 通过拆分Bucket的方式替代Chained Hashing中链表的无限增长
  • 每个Bucket上记录当前Bucket的有效位数
  • 当拆分发生时对当前Bucket中的元素进行重排序，并增加当前Bucket的有效位数
Linear Hashing

Extendible Hashing在拆分Bucket的时候需要锁住整个Hash表，待拆分完成后才会解锁，Linear Hashing做到了拆分时仅仅只需要锁住一个Bucket，而不是整个Hash表。

  • 通过一个spilt pointer指向下一个需要被拆分的Bucket，当哈希表发生overflow后，无论overflow的是否为指向的Bucket，都对该Bucket进行拆分
  • 拆分后在slot array中新增一个entry，并创建一个新的hash函数，对Bucket中元素进行重新分配
  • 在后续查询时，如果用原有hash函数查询后的结果在split pointer指向的Bucket之上，则用新的hash函数进行映射
  • 删除时，若Bucket为空，可以根据情况选择不缩减该Bucket或者按照拆分操作的逆操作流程，缩减对应Bucket

之所以可以不对当前overflow的Bucket进行拆分的原因在于，如果overflow一直发生，split pointer一定会指向所有发生overflow的Bucket，如果pointer指向了最后一个slot，就会删除第一个hash函数，并回到最初的slot位置
