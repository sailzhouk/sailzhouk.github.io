Database Storage

Posted on 2022-04-15

Disk-Oriented Architecture

计算机的物理存储可以按Volatile和Non-Volatile进行区分

Volatile:

  • 断电后数据丢失–>不能保存持久化的数据
  • 随机存取
  • 字节寻址

Non-Volatile:

  • 断电后数据不会丢失
  • 顺序存取–>按照顺序进行存储
  • 按照Block进行寻址

我们常用Memory（Volatile）和Disk（Non-volatile）来区分计算机的物理存储

课程中提到有Non-volatile Memory的存在，基于这种类型的存储或可构建出全新的DBMS

DB中的大量数据需要存储在Disk中，数据在Memory和Disk之间的移动会耗费大量的资源，DB需要精心设计存储结构保证读写操作的高效。

Why not os？

数据以文件的形式存储在磁盘中，一个文件中的每个页在被读取时需要被加载到内存中，即从虚拟内存加载到物理内存中。DB在设计时往往不使用OS自带的内存管理调用，如mmap，而是根据自己的需求设计合适的“disk-memory”数据交换策略。

在并发请求情况下，系统调用可以满足对于read请求的要求，但对于write请求而言，由于OS并不理解DB层面上的高级语义，从OS的视角来看，其仅仅只执行相应的读写指令，而并不知道这些指令对应的数据是什么样的，以及我们是怎样请求这些数据的。对于write请求，系统调用并不知道哪些page应该在另一些配置之前被刷到内存中。

DB可以做到更好的：

  • Flushing dirty pages to disk in the correct order.
  • Specialized prefetching.
  • Buffer replacement policy.
  • Thread/process scheduling.

数据在Disk中的存储形式

Problem #1: How the DBMS represents the database in files on disk.

File Storage

DB通过Storage Manager去管理其存储的File，一个File由多个Page组成，Storage Manager需要：

  • 追踪对每一页上数据的读写
  • 寻找空余的存储空间

每一个Page由唯一的标识符进行标记，表示固定大小的连续数据，Page存储的内容分为：tuples、logs、meta-data、indexes…一般一个page不会混合存储多种类型的数据（为了最大程度地从crash中恢复）

对于DB来说，Page有三种：

  • Hardware Page(4KB)
  • OS Page(4KB)
  • DB Page(512B-16KB)

不同厂家对DB大小的设计不同，这是一种取舍，过小的Page容易导致目录页过大，从而使得cache没办法存储下目录页（类似CPU的TLB），而大的Page又会需要很多的额外操作来保证对一个Page的写入是原子的，计算机只能保证对一个Hardware Page(4KB)的写入是原子的，因此大于其大小的Page需要额外增加逻辑保证写入的原子性，也会在某种程度上降低性能。

DB中的文件用Heap File的形式进行存储

A heap file is an unordered collection of pages where tuples that are stored in random order.

  • Create / Get / Write / Delete Page

  • Must also support iterating over all pages.

Heap File有链表实现（不采用）/页目录实现两种实现方式

页目录实现：

Directory中包含Page的空闲信息，指针等内容

Page Layout

每个Page有Header和Data部分组成

Header中保存信息：

  • checksum
  • page size
  • DBMS version
  • transaction visibility
  • compression information(gzip/etc..)

对于Page中保存数据即Tuples，一般可以有两种思路：

  • 维护一个num tuples，每次将新的tuple添加到最后（不适用删除场景，并且tuple的长度也是固定的）
  • 从前向后维护slots，从后向前存储数据，slots中存储tuple的首地址（tuple长度可变）
    • 在这种情况下header只需保存已用的slots信息以及最后一个已用的slot的首地址

Tuple Layout

tuple由一串连续的bytes组成，根据schema中属性的长度，解析出tuple中每个属性的值，schema中的meta-data也无需存储在tuple中

tuple=header+data，其header部分主要存储：

  • visibility info (for concurrency control)
  • bitmap for null values

对于一个tuple，其id可以用pageid+slotsid进行表示，不同的DB有不同的实现方式

denormalize tuple data：

一般不同table的tuples都存储在不同的page上，而当遇到比如join操作时，数据库会将不同table的tuple在物理存储上连接到一起，以优化后续操作的性能

muduo-学习Acceptor

Posted on 2021-06-02 In muduo学习

概述

muduo的Acceptor类实现了接收新的连接请求，并且通过回调创建新连接的功能。

  • setNewConnectionCallback()：由上层TcpServer调用，设置接收到新的连接连接请求时的回调函数。
  • listen()：用于启动监听
  • handleRead()：注册到acceptChannel_中，用于事件分发
  • idleFd_：用于解决busyloop

连接建立与busyloop处理

连接建立

在acceptChannel_触发后，EventLoop通过handleEvent()执行在其上注册的回调函数Acceptor::handleRead()，该函数只能在所属IO线程中调用，首先获得对等方的文件描述符，在其有效的情况下调用newConnectionCallback_创建连接。

busy-loop

当连接达到文件描述符的上限时，此时由于没有新的连接套接字用于建立新连接，则连接请求会一直在accept队列中，所以acceptChannel_上的读事件就会一直被触发造成busy-loop。

muduo提供的解决方案是在构造函数中创建一个只读的空洞文件描述符，用idleFd_保存，一旦文件描述符不够用，就会关闭idleFd_，然后用idleFd_接收新连接请求，然后再一次关闭它，最后仍然用idleFd_保存一个空洞文件描述符，相当于在新连接到达上限后muduo用idleFd_忽略了新连接，避免读事件一直触发。

muduo中还解决了一个可能会造成busy-loop的问题，因为muduo采用水平触发，如果在写入事件完成后，不立即取消关注写入事件，就会造成busy-loop。

muduo-tcpserver

Posted on 2021-06-02 In muduo学习

概述

TcpServer的功能主要是控制acceptor接收的连接包括

  • 完成创建新连接的步骤
  • 为新连接分配线程，
  • 删除连接

用户在使用TcpServer这个类的时候，只需要用基于对象的思想，将该类作为一个成员，通过该类提供的接口设置好回调函数，直接调用start()。

构造与析构

构造函数：

创建acceptor和threadPool，设置默认的回调，将newConnection()注册到acceptor中，析构时让connection所属的EventLoop来完成连接的销毁任务

启动Server，创建连接

当用户设置好连接回调和消息到达回调，调用start()后，一个TcpServer就得以启动，它通过acceptor调用其listen()函数，启动对serverSocket_的监听，同时也会关注channel_的读事件，新连接到达后，acceptor会回调newConnection()函数来创建新连接。

newConnection()首先从线程池中按照round-robin的方式取出一个EventLoop，同时自动设置连接的名字作为connectionMap的键值。最后创建连接，用智能指针来管理它，并且向连接对象注册回调，调用EventLoop执行连接建立完成回调。

muduo-学习TcpConnection

Posted on 2021-05-31 In muduo学习

概述

muduo用TcpConnection来表示一个Tcp连接的抽象概念，代表一个服务器-客户端之间的连接。当监听socket接收到了对等方的连接请求后，通过::accept系统调用，得到了对等方的文件描述符，之后两者间就建立了连接，muduo将连接的概念抽象为一个单独的类，提供了如下的功能：

  • 保存了服务器socket地址信息（ip+port）和对等方socket地址信息，便于查询
  • 封装了一系列的函数用于向连接对应的文件描述符，进行读/写操作
  • 提供了连接建立、连接关闭、消息到来、消息发送完毕的回调函数，处理好“三个半事件”
  • 维护了自身的状态（未连接、正在连接、已连接、正在关闭连接）

通过TcpConnection类，可以大大降低服务器编程的难度，使得程序的逻辑更加清晰。

这个类是一个内部类，不向客户开放，由于其模糊的生命期（某个客户端断开连接，但由于不知道有没有其他地方仍然在使用它，因此不能直接delete）其生命周期由shared_ptr进行管理。

三个半事件

TcpConnection首先在构造时，向对应的channel注册了关注事件的回调函数，从而在对应channel响应时调用回调来处理TcpConnection的事件（三个半事件+highWaterMark）

连接建立

连接建立的操作主要由Acceptor+TcpServer实现，当channel回调Acceptor的回调时，进行连接的建立操作，而Acceptor的回调由TcpServer进行设置，主要就是创建新的TcpConnection对象，并执行TcpConnection::connectEstablished回调，详细内容在TcpServer中进行介绍。

TcpConnection::connectEstablished()改变TcpConnection的状态，并且绑定channel_，提升其生命周期，监听channel_上的读事件，并执行connctionCallback_()（由用户设置）

断开连接

上图是TcpConnection断开连接的时序图，就像TimerQueue一样，TcpConnection也借助Channel，来完成关注事件的处理，当Channel上出现事件后，通过Channel::handleRead()的返回值调用不同的回调函数，当返回值要关闭TcpConnection时就会调用通过TcpConnection::handleClose()调用其closeCallback_，该回调由TcpServer设置。

消息到达

由channel_回调TcpConnection::handleRead()

用inputBuffer_读取数据，执行messageCallback_（由用户设置），否则关闭连接

TcpConnection::handleError() 用于处理错误事件，仅仅是向日志中记录错误事件不做更多的操作

消息发送完毕+高水位

  • TcpConnection通过send和sendInLoop接口，来发送数据，发送数据的实际操作由所属的EventLoop执行。

  • sendInLoop会尝试一次发送数据，如果一次就发送完，就会直接退出，否则会将数据存储到outputBuffer中，关注writable事件。

  • 在关注writable事件后，一旦可写就会调用handleWrite写入数据，一旦outputBuffer为空，就立即调用writeCompleteCallback_，取消关注writable事件，防止busy loop。

  • sendInLoop在将数据放入到outputBuffer时，如果数据量过大，超过了高水位highWaterMark，此时就会触发高水位回调，停止读取数据，随后会在writeCompleteCallback中恢复读取数据

muduo-多线程的EventLoop

Posted on 2021-05-28 Edited on 2021-05-31 In muduo学习

muduo通过EventLoopThread和EventLoopThreadPool这两个类来提供多线程的EventLoop接口。

EventLoopThread

EventLoopThread本质上是一个线程类，行为与使用Thread类时所要定义的类似，需要一个回调函数定义该线程的行为，同时将对应的EventLoop传回给主线程。

EventLoopThread包含一个Thread和EventLoop成员，这体现了muduo设计时基于对象的思想。

构造函数主要是创建线程，初始化ThreadInitCallback回调

析构函数退出loop，可能与ThreadInitCallback冲突

startLoop()启动线程，用condition锁等待新线程中EventLoop对象创建完成，然后返回EventLoop对象，EventLoopThread中的EventLoop成员loop_由条件变量和互斥锁来进行保护，条件变量等待新线程创建EventLoop；

新线程启动后，首先执行初始回调函数，随后创建EventLoop对象，启动其loop循环，循环退出后将loop_设置为空。

EventLoopThreadPool

分别数组保存所有的EventLoopThread对象和EventLoop对象，通过start()函数启动，随后按照round-robin的方式从ThreadPool中依次取出EventLoop。

start()依次在堆上创建新的EventLoopThread对象，将其放入threads_数组中，接着依次调用它们的startLoop()方法，将返回的EventLoop存入loops_中。

getNextLoop()按照roundrobin()的方式依次得到loops_中的一个EventLoop对象。

muduo-学习Eventloop

Posted on 2021-05-27 In muduo学习

EventLoop框架

one loop per thread表示一个线程只能有一个EventLoop对象，EventLoop通过threadId_来记录所在线程，EventLoop的核心方法是loop()，它会一直处于循环状态。EventLoop还提供了一系列的接口：

  • 用runInLoop()在线程间调配任务（让IO线程也能执行一些回调函数）；
  • 对定时器做了更高层次的封装（runAt()、runAfter()、runEvery()）；
  • 使用eventfd来唤醒阻塞中的线程；
  • 作为IO multiplexing事件分发的“桥梁”（updateChannel()、removeChannel()、hasChannel()）;
  • 提供了quit()退出事件循环；

EventLoop的事件分发

EventLoop构建起了整个reactor最核心的事件分发机制，这通过loop()函数实现，该函数必须由IO线程调用，activeChannels是一个传出参数。在Poller从poll()返回后，会依次执行activeChannels中的所有事件回调。EventLoop通过Poller将事件写入给相应的Channel，由channel自身执行事件回调，这也符合reactor的思想，Channel的成员函数都只在IO线程中调用。

当channel需要更新其关注的事件时，就会通过EventLoop来调用相应的poller更新事件。

上图为事件分发过程的时序图，EventLoop调用poll()，有事件触发时返回，然后依次调用对应Channel的事件回调；

EventLoop退出loop循环

EventLoop通过设置quit_来退出循环，如果不在IO线程则去唤醒所属EventLoop，如果在IO线程，说明此时未被阻塞，则直接退出

这里的问题在于EventLoop有析构的可能，因此可能会访问无效的对象，可以用锁来保护。

EventLoop在线程间调配任务

EventLoop通过runInLoop()和queueInLoop()来实现在线程间调配任务，使得IO线程除了poll的事件循环，也能执行一些回调函数。

例如在TimerQueue的cancel()成员函数，将cancel的具体操作用loop来进行回调，从而在其IO线程执行该成员函数，保证线程安全性。

runInLoop()和queueInLoop()

如果在loop所属IO线程中，则直接调用cb()，因为此时也表明线程没有被poll阻塞，所以可以直接执行cb()

否则，就将cb()添加到pendingFunctors中，然后在两种情况下唤醒loop，分别是：

  • 非所属loop线程
  • 正在执行阻塞的functors（functors()中也可能会调用queueInLoop()）

对照loop()循环的相应代码，可以发现，只有在执行事件回调的时候才无需唤醒。

通过eventfd进行唤醒

要唤醒阻塞在poll()的IO线程，可以通过pipe来做，但muduo选择的是eventfd，这是一种比pipe更高效的事件通知机制，eventfd节省了一个file descriptor的资源，并且buffer也是定长的8个字节，最重要的是eventfd完美适配reactor的结构，就像其他的监听文件描述符一样，唤醒epoll_wait这样的异步IO。

doPendingFunctors()

先将回调数组swap到临时数组中，再依次调用functor，减小了临界区的大小，同时也避免再functor中调用queueInLoop发生死锁。

EventLoop的定时器

上图为muduo中一次事件循环中各种回调发生的顺序，可以看到在muduo中，timers与IO事件处理放在了一起，这主要是timerfd的作用，类似于eventfd，他们提高了整个reactor代码结构的一致性。而传统的方法则将timers放在事件循环中的某一步。

上图为TimerQueue的回调用户设置的回调函数onTimer()的时序图，可以看到，对定时器的处理相当于是一个“IO事件”，timerQueue中保存了对应的channel，在触发后也是按照对应的回调函数去处理，TimerQueue通过getExpired得到到期的定时器，然后依次执行它们的用户回调，

EventLoop在TimerQueue的基础上封装了更加简洁的runAt()，runAfter()，runEvery()等方法，代码逻辑也很简单，就是向timerQueue中添加对应的定时器。

但是addTimer()函数只能在IO线程中调用，所以runAt()这些函数并不是线程安全的，于是就可以用之前介绍的方法，提供addTimerInLoop()函数，借助EventLoop的runInLoop()方法，将定时器的调用移动到IO线程来，避免加锁。

muduo-学习Poller以及EPollPoller

Posted on 2021-05-26 Edited on 2021-05-28 In muduo

muduo中有一个Poller类用于提供IO多路复用的功能，Poller是一个基类，其子类有EPollPoller和PollPoller，我们这里只看EPollPoller的实现，我们后面简单地用Epoller来代指它。

前面在Channel的学习中提到，Channel、Epoller以及EventLoop三者是相关联的，Channel可以通过其所属EventLoop来找到监听它的Epoller，Epoller本身并不包含Channel，EventLoop就像是连接它们的桥梁。

因此EPoller保存一个EventLoop指针，那么这个类又相对于系统调用封装了哪些功能呢？

分发/传回响应事件

首先由于muduo的one eventloop per thread+thread pool这种模式，每个Channel上发生的事件最好都由一个专门的线程去处理，因此Epoller不能仅仅只返回包含一个响应的事件的数组，而要能够将响应的事件分发出去，这可以通过fillActiveChannels来实现，因为Channel内部有一个revents成员，因此，Epoller只需要将响应的事件注册给Channel，同时将所有的响应Channel传出给EventLoop，由EventLoop来让对应Channel执行响应的事件。

更新监听Channel信息

对应epoll_ctl这个系统调用，用于增加、删除、更新对应Channel的监听事件

我们首先来看Poller类

Poller类定义了的方法只有hasChannel，这个类也只能由loop所在线程调用，因此有assertInLoopThread()。

实现基类的纯虚函数

poll用epoll_wait监听事件，muduo这里使用的是水平触发，作者认为这样容易通过strace检查程序的行为，poll调用fillActiveChannels传回activeChannels，以及注册channel的**revents_**，关于出错处理，当errno不为EINTR时会在日志记录错误，但并不会退出程序

fillActiveChannels这里用了传出events_数组中epoll_event结构的特点，每个epoll_event的data的ptr成员被设置为对应的channel（见update()），从而可以通过传出events_来填充channels_数组

updateChannel由loop所在线程调用，对channel的状态进行了区分，将channel对应的fd添加到**epollfd_ **中。

removeChannel则是将channel从**channels_ 中移除，在移除前需要确保其状态为kDeleted，否则要先向epollfd_**更新状态

update使用::epoll_ctl来更新关注fd，增加了对错误状态的封装，如果是删除操作出错则仅仅只是error记录响应日志信息，其他操作出错会直接退出。

muduo-学习Channel Class

Posted on 2021-05-14 Edited on 2021-05-26 In muduo

muduo的Channel Class是socket的扩展，提供了对socket进行处理的一些接口，有一个对应于socket的文件描述符fd_ , 但其并不负责管理它的生命周期，即不会关闭该文件描述符，其内部扩充了用于IO复用epoll/poll的事件成员events_ /revents_，每个Channel只属于一个EventLoop，EventLoop是Channel和Epoller沟通的桥梁。

Channel的功能：

  • 设置文件描述符对应的关注事件（用于epoll）
  • 向epoll中注册监听事件（通过所属EventLoop找到对应的epoll）
  • 注册不同事件的回调函数
  • 处理事件，根据返回的事件类型调用相应的回调函数

也就是说Channel负责几乎一切其所属文件描述符上的动作，对比之前实现的版本，当事件发生时我们是在Epoll\Poll\Select中执行相应事件，这与这些类自己的抽象概念并不符合，也缺乏灵活性，现在将这些动作抽出来就显得合理许多，同时这也为后续muduo的one loop per thread + thread pool模型奠定了基础，只有将文件描述符上对应的事件处理\设置等功能抽象出来，才能实现在不同的Event Loop中做不同的任务。

许多的传输层都具有带外数据（也称为 经加速数据 ）的概念，想法就是连接的某段发生了重要的事情，希望迅速的通知给对端。这里的迅速是指这种通知应该在已经排队了的带内数据之前发送。也就是说，带外数据拥有更高的优先级。带外数据不要求再启动一个连接进行传输，而是使用已有的连接进行传输。

muduo-学习Buffer Class

Posted on 2021-05-14

non blocking网络编程中需要buffer

为什么需要non blocking

non blocking可以避免被read或write这种IO系统调用所阻塞，从而一个线程可以服务于多个连接，否则一旦一个线程被阻塞，它就无法处理其他线程的IO事件

output buffer

output buffer用于发送数据，因为TCP流量控制的影响，程序不能一次性将所有数据发出，为了不在原地等待所有数据发送完成，可以将数据发送到output buffer中。

网络库通过注册POLLOUT事件，托管程序所有的发送操作，一旦socket可写，就发送数据，当数据写完后就立即取消关注POLLOUT，防止busy loop（由于LT触发导致，只要socket可写，就会一直触发，但此时又没有数据需要写入）

input buffer

TCP是无边界的字节流协议，所以一次接收到的既可能是多条消息也可能是一条不完整的消息，也就是所谓的粘包现象，因此需要input buffer来暂存数据，通过应用层调用来判断消息是否完整，然后将完整的消息取出。input buffer从内核缓冲区读取数据时应尽可能一次读取所有的数据，减少POLLIN事件的触发。

buffer的构造函数仅仅只是通过给定的初始大小来构造一个buffer，将readIndex和writeIndex都指向kCheapPrepend（为初始前置区域的大小）
